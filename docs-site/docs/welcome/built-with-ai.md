# Built with AI, on purpose

This entire codebase was written by Claude (Anthropic's AI). Not as a gimmick, not as a shortcut: as a deliberate experiment in what AI-assisted development actually looks like when you treat it seriously.

## What that means in practice

Every line of code was generated by Claude Code. Every architectural decision was made by a human (me). The process looks like this:

```
Research (Claude.ai) → "what exists, how does it work?"
    ↓
Decision (me) → "we'll use X with Y architecture"
    ↓
Implementation (Claude Code) → code, tests, CI
    ↓
Problem (I test it) → "this doesn't work"
    ↓
Research → Implementation → Problem → repeat
```

This cycle ran 20+ times for the music pipeline alone. 9 attempts for the video assembly pipeline, each one failing in a different way, each revealing more about how FFmpeg actually works vs. how the documentation says it works. The assembly odyssey (2 months of debugging encoder non-determinism) is documented in the dev history with full transcripts.

## What's novel

**The development pattern, not the output.** AI writing code isn't new. What's interesting is the workflow:

1. **Research happens in conversation, not in code.** Before any feature gets built, I spend 3-6 rounds with Claude.ai (not Code) doing pure research: what exists, what are the trade-offs, what's the state of the art. The music pipeline started with "what music APIs exist?" and went through VRAM sizing, K8s API design, beat-aligned crossfading, and quality metrics before a single line of code was written.

2. **The human is the debugger, not the coder.** The hardest bug in the project (frame discontinuities in the assembled video) was diagnosed by me looking at individual frames side-by-side, noticing pixel differences between encoding passes, and realizing the root cause was encoder non-determinism across separate FFmpeg invocations. Claude's 8 previous fix attempts were all wrong because they were based on different hypotheses. The fix required understanding what was happening at the pixel level.

3. **Quality gates force discipline.** 17 CI checks, 1900+ tests, mutation testing, complexity limits, file length caps, architectural boundary enforcement. These exist because AI-generated code without guardrails is just fast garbage. The CI pipeline catches the "looks right but isn't" code that AI produces when it's confidently wrong.

4. **Every feature follows research-first, not code-first.** Audio ducking (lowering music during speech): 3 research rounds on stem separation before coding. Face-aware cropping: research on OpenCV cascades vs. Vision framework vs. CUDA-accelerated detection before picking an approach. Trip detection: GPS clustering algorithms, overnight detection heuristics, segment distribution math, all researched before implementation.

## The numbers

- **1900+ tests** across unit and integration tiers
- **17 CI gates** (lint, format, types, complexity, security, dead code, architecture, dependencies, more)
- **80+ source files**, all under 800 lines (enforced)
- **Composition over inheritance**: zero mixins, zero multiple inheritance
- Every mock has a `# WHY:` comment explaining what external boundary it replaces

## Why mention this at all

Because most "built with AI" projects are either:
- Disclaimers ("AI helped, don't blame us")
- Marketing ("look how fast we shipped")

Neither is interesting. What's interesting is the honest answer: AI writes the code, a human makes the decisions, and without serious quality infrastructure, the result is garbage regardless of who typed it. The 17 CI gates aren't decoration: they caught real bugs that Claude introduced confidently.

This project exists because the development methodology works, not because the AI is magic.
